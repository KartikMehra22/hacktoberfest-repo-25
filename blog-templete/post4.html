From Docker to OCI images: a practical migration

Move your images to OCI-compliant layouts and keep your pipelines sane.

By Alex Kumar Jun 16, 2025 Cloud Containers 8 min read

Why OCI matters

The Open Container Initiative (OCI) standardizes image formats and runtimes so containers can be built, shared, and run anywhere—without vendor lock-in. Migrating to OCI ensures long-term compatibility with modern registries and orchestrators like Kubernetes.

Docker vs OCI layout

Docker image manifests were historically proprietary, with layered JSON structures that didn’t conform to OCI’s strict specification. OCI images, however, store content as digest-addressed blobs and define a clear schema for configuration and layers. 

# Docker image structure
manifest.json
├── config.json
└── layer.tar

# OCI layout
blobs/
├── sha256:...
index.json
oci-layout

How to migrate

  1. Use docker buildx build --output=type=oci to export directly as OCI.
  2. Convert existing images using skopeo copy docker://... oci:....
  3. Push to an OCI-compliant registry like GHCR or Harbor.

Benefits

  • Interoperability across multiple runtimes (containerd, Podman, etc.).
  • Immutable content-addressed layers → safer caching.
  • Standardized metadata schema → easier automation and signing.

Example: converting an image

# Convert and inspect OCI image
skopeo copy docker://nginx:latest oci:nginx-oci:latest
ls nginx-oci/
cat nginx-oci/index.json | jq

Final thoughts

Moving to OCI images isn’t just about compliance—it’s about reliability and future-proofing. Your CI/CD pipelines become cleaner, and you eliminate hidden dependencies tied to Docker’s proprietary layers.
